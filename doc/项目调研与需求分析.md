# 项目调研与需求分析

## 一、项目名称

基于 Modbus 与 MQTT 的机柜多源状态监测与远程运维系统

## 二、项目背景

在校园机房、小型企业机柜、社区弱电间和网络设备间等场景中，机柜通常承担交换机、路由器、网关、电源模块等关键设备的集中放置任务。机柜内部温度、湿度、烟雾、光照和供电状态会直接影响设备运行稳定性。如果缺少持续监测能力，常见问题包括温度过高导致设备宕机、烟雾或可燃气体异常未及时发现、机柜被打开后无法及时感知、人工巡检周期长且效率低等。

传统巡检方式主要依赖人工定时查看，存在响应滞后、记录不连续、异常难追溯的问题。本项目希望利用嵌入式采集、RS485 总线通信、MQTT 上云和 Web 可视化技术，构建一个面向小型机柜场景的低成本状态监测与远程运维系统。系统在本地完成实时采集与告警联动，并将关键数据上传到 OneNET 云平台和后端服务，为后续远程查看、报警记录、阈值配置和设备控制提供基础。

## 三、市场与应用分析

### 3.1 目标场景

本项目主要面向以下应用场景：

| 场景 | 典型问题 | 系统价值 |
| --- | --- | --- |
| 校园网络机柜 | 巡检人员少、设备分散 | 降低人工巡检压力，及时发现异常 |
| 小型企业机房 | 缺少专业动环监控系统 | 用较低成本实现核心环境监测 |
| 社区弱电间 | 空间封闭、环境不可控 | 提供温湿度、烟雾等安全监测 |
| 实验室设备柜 | 设备频繁调试、状态变化快 | 方便教学演示和工程实践 |

### 3.2 同类方案分析

商用机房动环监控系统通常功能完整，能够覆盖温湿度、水浸、门禁、电量、视频和联动控制，但设备成本和部署复杂度较高。对于课程实践、小型机柜和轻量化监控场景，完整商用系统并不一定合适。

本项目采用 STM32 主控、STM32F103 从机、RS485/Modbus 总线、RK3568/MQTT 和 Spring Boot 后端，具有结构清晰、便于调试、网关扩展能力强和可逐步完善的特点。虽然当前功能规模小于商用系统，但已经覆盖机柜环境监测的核心链路，适合课程项目和小规模应用验证。

### 3.3 可行性结论

从硬件可获得性、软件实现难度、协议成熟度和课程交付要求来看，本项目具备较高可行性。STM32、DHT11、MQ-2、光敏电阻、RS485 模块和 RK3568 均属于常见开发硬件，资料丰富；Modbus RTU 和 MQTT 协议成熟稳定；Spring Boot 便于快速构建接口和监控页面。因此项目能够在有限周期内完成从采集、通信、控制到展示的完整闭环。

## 四、项目目标与范围

### 4.1 项目目标

1. 完成机柜环境多源数据采集，包括光照、温度、湿度和烟雾浓度。
2. 完成基于 RS485 的 Modbus RTU 主从通信，实现主控对多个从机的统一轮询。
3. 完成本地告警判断，根据阈值驱动蜂鸣器、LED 和风扇执行对应动作。
4. 完成基于 MQTT 的数据上传，使监测数据能够进入 OneNET 云平台或后端服务。
5. 完成 Spring Boot 后端基础接口和监控页面，实现设备状态查看。
6. 预留电压、电流、门磁、继电器等后续扩展接口。

### 4.2 项目范围

本阶段项目范围以“可采集、可通信、可告警、可展示”为核心：

| 范围类别 | 包含内容 | 不包含或后续扩展内容 |
| --- | --- | --- |
| 嵌入式采集 | 光敏电阻、DHT11、MQ-2 | 高精度工业传感器 |
| 总线通信 | RS485、Modbus RTU | CAN、LoRa 等其他总线 |
| 本地控制 | LED、蜂鸣器、风扇 PWM | 复杂联动策略和自动断电保护 |
| 上云通信 | RK3568、MQTT、OneNET 接入 | 历史数据缓存和多协议转发 |
| Web 后端 | 登录、状态缓存、基础接口 | 完整历史数据库和多租户管理 |

## 五、技术分析

### 5.1 技术路线

系统采用“从机采集、主控汇聚、本地联动、网络上传、后端展示”的技术路线：

1. 传感器节点由 STM32F103C8T6 负责采集数据。
2. 每个从机将采集值映射为 Modbus 保持寄存器。
3. STM32F429 主控通过 RS485 总线轮询各从机。
4. 主控根据阈值进行本地告警判断，并控制蜂鸣器、LED 和风扇。
5. 主控将数据发送给 RK3568，由 RK3568 使用 MQTT 协议上传数据到 OneNET。
6. Spring Boot 后端接收或查询设备状态，并通过页面展示。

### 5.2 关键技术

| 技术点 | 作用 | 可行性说明 |
| --- | --- | --- |
| STM32 裸机开发 | 完成主控与从机程序 | 仓库中已有多个 Keil 工程 |
| ADC 与单总线采集 | 读取光照、烟雾、温湿度 | 传感器驱动结构清晰 |
| RS485/Modbus RTU | 实现多从机通信 | 协议简单可靠，适合工业场景 |
| PWM/GPIO 控制 | 控制风扇、LED、蜂鸣器 | MCU 外设资源满足要求 |
| RK3568/MQTT | 完成边缘网关转发和数据上云 | 计算资源更充足，便于扩展缓存、协议转换和边缘处理 |
| Spring Boot 3 | 构建后端接口 | 便于快速开发和部署 |

### 5.3 风险分析与应对

| 风险 | 表现 | 应对措施 |
| --- | --- | --- |
| RS485 通信不稳定 | 从机响应超时、CRC 错误 | 统一波特率、地址和接线，增加超时重试 |
| 传感器数值波动 | 数据跳变影响告警判断 | 增加有效范围判断和简单滤波 |
| MQTT 连接异常 | 数据无法上云 | 保证本地告警独立运行，联网模块支持重连 |
| 硬件调试周期紧 | 多模块联调复杂 | 先完成单节点测试，再进行系统集成 |
| 后端数据格式不一致 | 页面显示异常 | 统一数据字段和接口返回格式 |

## 六、功能性需求

### 6.1 数据采集需求

1. 系统应支持光照强度采集，并能够通过 Modbus 寄存器返回给主控。
2. 系统应支持温度和湿度采集，并能够分别保存最新温度值和湿度值。
3. 系统应支持 MQ-2 烟雾或可燃气体浓度采集，并输出相对浓度值。
4. 系统应支持从机状态标识，便于主控识别设备正常或故障状态。

### 6.2 通信需求

1. 主控应作为 Modbus RTU 主站，周期性轮询地址为 `0x01`、`0x02`、`0x03` 的从机。
2. 从机应支持读取保持寄存器功能，返回对应传感器数据。
3. 主控应对 Modbus 响应帧进行 CRC 校验，避免错误数据进入业务判断。
4. 主控应将采集数据发送给 RK3568，RK3568 应将数据封装为 MQTT 消息并上传。

### 6.3 告警与控制需求

1. 系统应支持正常、运行警报、严重警报三个状态等级。
2. 当任一监测参数超过阈值时，系统应进入运行警报状态。
3. 当两个及以上参数超过阈值，或烟雾浓度达到严重阈值时，系统应进入严重警报状态。
4. 系统应根据告警等级控制 LED 指示灯和蜂鸣器。
5. 系统应根据温度阈值控制风扇转速或启停。

### 6.4 后端与展示需求

1. 后端应提供设备状态查询接口，返回最新温度、湿度、光照、烟雾和告警状态。
2. 后端应具备基础登录或访问控制能力，避免任意用户操作设备。
3. 页面应能够展示实时监测数据、连接状态和执行器状态。
4. 后端应为后续报警记录、历史曲线和远程控制保留扩展空间。

## 七、非功能性需求

| 类别 | 需求描述 |
| --- | --- |
| 稳定性 | 网络异常时，本地采集和告警仍能独立运行 |
| 实时性 | 采集与告警响应应达到秒级以内 |
| 可扩展性 | 从机地址、寄存器和后端字段应便于扩展 |
| 可维护性 | 传感器驱动、通信协议、告警逻辑和后端服务分模块实现 |
| 成本控制 | 优先选用通用开发板、常见传感器和开源框架 |
| 易用性 | Web 页面应清晰展示核心状态，便于演示和巡检 |

## 八、需求优先级

| 优先级 | 需求 |
| --- | --- |
| 高 | 传感器采集、Modbus 通信、本地告警、风扇联动 |
| 中 | MQTT 上传、OneNET 接入、后端状态展示 |
| 低 | 历史曲线、报警记录、门磁、电压电流、多协议网关扩展 |

## 九、需求分析结论

本项目围绕机柜环境安全和远程运维展开，背景明确、目标清晰、技术路线可行。当前阶段重点实现光照、温湿度、烟雾三类数据采集，完成主从通信、本地告警和基础上云展示；后续可继续扩展电源监测、门禁监测、历史数据存储和边缘网关能力。整体方案能够满足课程项目对背景分析、市场分析、目标范围、技术可行性、功能需求和非功能需求的要求。
